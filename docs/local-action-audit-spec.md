# Local Action Audit Spec (Phase 13F — future contract, not runtime code)

> **Status:** Architecture / contract baseline only. Phase 13F is an
> implementation candidate, pending review, **not locked**. **No audit record
> is generated by Phase 13F** — nothing executes. This document defines the
> future durable audit record so a later Local-mode phase produces accountable,
> human-inspectable records.

## 1. Audit record fields

| Field | Required | Meaning |
|---|---|---|
| `schemaVersion` | Yes | Contract version, e.g. `"1.0"`. |
| `auditId` | Yes | Unique id for this audit record. |
| `requestId` | Yes | The originating action request's `requestId`. |
| `action` | Yes | The allowlisted action name. |
| `target` | Yes | The object the action targeted (stable ids). |
| `workspace` | Yes | Workspace identity `{ id, root, slot? }`. |
| `storageTarget` | Yes | Storage target descriptor `{ type }`. |
| `preconditionsEvaluated` | Yes | The preconditions checked and their outcomes. |
| `decision` | Yes | `"accepted"`, `"rejected"`, `"deduplicated"` (future). |
| `result` | Conditional | Outcome once known: `"completed"`, `"failed"`, or `null` while pending. |
| `createdAt` | Yes | ISO-8601 timestamp the record was created. |
| `completedAt` | Conditional | ISO-8601 timestamp of completion; `null` while pending. |
| `evidenceRefs` | Yes | Read-only references relevant to the action. |
| `idempotencyKey` | Conditional | Present for retry / rerun actions; ties duplicate submissions together. |
| `notes` | Yes | Free-text notes; may be empty string. |

## 2. Principles

- **No audit record is generated by Phase 13F.** The example audit fixture is
  a documentation artifact, not a generated runtime output.
- **Future acceptance is not the same as future execution success.** An audit
  record may show `decision: "accepted"` with `result: null` while the action
  is still pending; success is recorded only when `result: "completed"` and
  `completedAt` is set.
- **Async completion should produce durable action-result and audit state.**
  When a long-running action finishes, both the action result (in
  `action-results/`) and the audit record (in `audit/`) are updated durably.
- **Rejected / failed actions should still be auditable where appropriate.** A
  rejected action (e.g. failed precondition, deferred action name, duplicate
  idempotency key) may produce an audit record with `decision: "rejected"` or
  `"deduplicated"` and `result: "failed"` / `null`, so the accountability trail
  is complete.

## 3. Storage

Audit records live in the workspace `audit/` directory (see
`docs/local-mode-workspace-layout.md`). The directory is append-only and must
be included in backups. Audit records are human-inspectable JSON.

## 4. What Phase 13F does NOT do

No audit record is written, no `audit/` directory is created, and no runtime
code produces audit state. The example fixture is illustrative only.
