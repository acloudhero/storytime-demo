# Local Action DTO Spec (Phase 13F — future contract, not runtime code)

> **Status:** Architecture / contract baseline only. Phase 13F is an
> implementation candidate, pending review, **not locked**. **These are future
> contracts, not active runtime code.** Phase 13F implements no DTO. The JSON
> examples under `docs/examples/` are documentation fixtures only — they are
> never imported by runtime code, never generated by a running system, and
> never claim Phase 13F executed anything.

## 1. Request DTO

A future Local-mode action request is a versioned, typed object naming an
**allowlisted** action plus validated parameters. There is no free-form
command, shell, or SQL field — ever (see `docs/local-bridge-architecture.md`
§8).

### 1.1 Request fields

| Field | Required | Meaning |
|---|---|---|
| `schemaVersion` | Yes | Contract version, e.g. `"1.0"`. Mismatches are rejected. |
| `requestId` | Yes | Caller-assigned unique id for this request. |
| `mode` | Yes | Operating mode; future requests use `"local"`. |
| `action` | Yes | Allowlisted action name (see §3). |
| `target` | Yes | Object the action targets (run id / stage id / export, as applicable). |
| `workspace` | Yes | Workspace identity: `{ id, root, slot? }`. |
| `storageTarget` | Yes | Storage target descriptor: `{ type }` (e.g. `"local-disk"`). |
| `dryRun` | Yes | If `true`, validate without executing (future). |
| `requiresConfirmation` | Yes | If `true`, operator confirmation is required before execution. |
| `requestedAt` | Yes | ISO-8601 timestamp the request was constructed. |
| `operatorIntent` | Yes | Human-readable statement of intent. |
| `preconditions` | Yes | Array of precondition descriptors the bridge must evaluate. |
| `evidenceRefs` | Yes | Array of read-only references the operator should inspect. |
| `idempotencyKey` | Conditional | Required for `retry_failed_stage` / rerun; deduplicates execution. |
| `executionTiming` | Yes | `"sync-validation-only"` or `"async-long-running"`. |

### 1.2 Target sub-object

`target` carries only stable ids — e.g. `{ "runId": "...", "stageId": "..." }`
for a retry, `{ "runId": "..." }` for an inspect, `{ "exportPath": "..." }` for
a refresh. No absolute private user paths; demo-safe / workspace-relative
references only.

### 1.3 Workspace / storageTarget sub-objects

- `workspace`: `{ "id": "...", "root": "<workspace-relative or demo-safe>",
  "slot": "active" }`.
- `storageTarget`: `{ "type": "local-disk" | "ram" | "usb" | ... }`. No
  credentials, ever (see `docs/storage-targets-architecture.md`).

## 2. Response DTO

### 2.1 Response fields

| Field | Required | Meaning |
|---|---|---|
| `schemaVersion` | Yes | Contract version. |
| `requestId` | Yes | Echoes the request's `requestId`. |
| `accepted` | Yes | Whether the request was accepted for processing. **Acceptance is not success.** |
| `status` | Yes | `"accepted"`, `"rejected"`, `"validated"`, `"completed"`, `"failed"` (future). |
| `result` | Conditional | Present only on a completed action (future); otherwise `null`. |
| `errors` | Yes | Array; empty when none. |
| `warnings` | Yes | Array; empty when none. |
| `exportRefreshRequired` | Yes | Whether a deterministic export refresh is required after this action. |
| `auditRecordRef` | Conditional | Reference to the durable audit record, when one exists (future). |
| `actionRequestId` / `jobId` | Conditional | **Required for async accepted actions.** The handle the frontend uses to learn status later. |

### 2.2 Acceptance is not success

A response with `accepted: true` and `status: "accepted"` means the request was
queued / dispatched, **not** that it succeeded. For long-running actions the
frontend must read later status from the refreshed export / read model or an
explicit result lookup, keyed by `actionRequestId` / `jobId`. A response must
never set `status: "completed"` merely because it was accepted.

## 3. Allowlisted initial actions

Start with exactly three:

- **`retry_failed_stage`** — rerun a failed / blocked stage on an eligible run.
  The safest first real action. Requires `idempotencyKey`. Long-running →
  `executionTiming: "async-long-running"`.
- **`inspect_trust_envelope`** — read-only inspection of a run's governance /
  Trust Envelope context. Validation / read-only →
  `executionTiming: "sync-validation-only"`.
- **`refresh_export`** — regenerate the deterministic export / read model.
  Does not pretend to write the file in Phase 13F (the examples are
  illustrative only).

### 3.1 Deferred actions (NOT allowlisted initially)

- **`record_review_decision`** — deferred: highest-risk; mutates the Trust
  Envelope and gates downstream actions; needs the strongest audit + identity
  guarantees.
- **`regenerate_operator_report`** — deferred: content regeneration that
  writes files; lower risk but still a durable write, kept out of the first
  allowlist to keep the initial surface minimal.
- **`publish_episode`** — deferred: externally visible side effects.
- **`delete_artifact`** — deferred: destructive; needs compensation /
  confirmation design.
- **provider-sync actions** — deferred: touch credentials + network + external
  mutation; highest-risk class.

A future request example must use only an allowlisted action name; the test
`tests/test_local_mode_contract_examples.py` enforces that deferred actions do
not appear in the request example fixtures.

## 4. What Phase 13F does NOT do

No DTO type, class, parser, or validator is added to runtime code. The example
JSON files are documentation fixtures only. Nothing executes.
